<objective>
## Goal
Create the driver stock loading data model and admin/owner load creation workflow. Admin/owner can load products onto a driver (deducting from warehouse stock), print a load receipt, and view all loads.

## Purpose
Drivers currently sell from a shared warehouse stock pool with no per-driver accountability. This plan introduces driver-level stock allocation — when a driver starts their day, admin/owner loads specific products onto them. This creates an audit trail of what each driver carries and enables shift-close reconciliation in Plan 11-02.

## Output
- Migration 020: `driver_loads` + `driver_load_items` tables, `create_driver_load` RPC, RLS policies
- Driver load repository + Riverpod providers
- Create Load screen (admin/owner picks driver + products)
- Load Receipt screen with Bluetooth printing
- Load List screen (active/recent loads overview)
- Dashboard entry points for owner and admin
</objective>

<acceptance_criteria>

## AC-1: Atomic Load Creation
```gherkin
Given an admin/owner with products in warehouse stock (stock_on_hand > 0)
When they create a driver load with selected driver and products+quantities
Then a driver_loads record is created with status='active',
  driver_load_items are inserted for each product,
  products.stock_on_hand is decremented by loaded quantities,
  stock_movements are created with movement_type='load_out',
  and the new load ID is returned
```

## AC-2: One Active Load Per Driver
```gherkin
Given a driver already has an active load (status='active')
When admin/owner attempts to create another load for the same driver
Then the RPC rejects the operation with error 'driver_has_active_load'
  and no data is modified
```

## AC-3: Stock Validation on Load
```gherkin
Given a product has stock_on_hand = 5
When admin/owner tries to load 10 units of that product onto a driver
Then the RPC rejects with 'insufficient_stock'
  and no load is created (atomic rollback)
```

## AC-4: Load Receipt Printing
```gherkin
Given a driver load was successfully created
When admin/owner views the load receipt screen and taps Print
Then a Bluetooth receipt prints showing:
  driver name, all loaded products with quantities,
  loaded_by name, and timestamp
```

## AC-5: Load List View
```gherkin
Given multiple driver loads exist (active and closed)
When admin/owner navigates to the loads list screen
Then all loads are displayed with driver name, status badge,
  total items count, and date — sorted by most recent first
```

  <name>Task 1: Migration 020 — Driver load tables, RPC, RLS, indexes</name>
  <files>supabase/migrations/020_driver_loads.sql</files>
  <action>
    Create migration 020_driver_loads.sql with:

    **PREREQUISITE — ALTER stock_movements CHECK constraint:** <!-- audit-added -->
    The existing stock_movements table (migration 013) has:
    `CHECK (movement_type IN ('order_out', 'purchase_in', 'cancellation_restore', 'adjustment'))`
    This migration MUST ALTER that constraint FIRST to add 'load_out' and 'load_return':
    ```sql
    ALTER TABLE stock_movements DROP CONSTRAINT stock_movements_movement_type_check;
    ALTER TABLE stock_movements ADD CONSTRAINT stock_movements_movement_type_check
      CHECK (movement_type IN ('order_out', 'purchase_in', 'cancellation_restore', 'adjustment', 'load_out', 'load_return'));
    ```
    Without this, INSERT into stock_movements with 'load_out' will throw a CHECK violation.

    **Tables:**

    1. `driver_loads` table:
       - id UUID PK (gen_random_uuid)
       - business_id UUID NOT NULL
       - driver_id UUID NOT NULL (the driver receiving stock)
       - loaded_by UUID NOT NULL DEFAULT auth.uid() (admin/owner who created it)
       - status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'closed'))
       - opened_at TIMESTAMPTZ NOT NULL DEFAULT now()
       - closed_at TIMESTAMPTZ (set on shift close — Plan 11-02)
       - notes TEXT (optional note from admin)
       - created_at TIMESTAMPTZ NOT NULL DEFAULT now()
       - updated_at TIMESTAMPTZ NOT NULL DEFAULT now()

    2. `driver_load_items` table:
       - id UUID PK (gen_random_uuid)
       - load_id UUID NOT NULL REFERENCES driver_loads(id) ON DELETE CASCADE
       - product_id UUID NOT NULL
       - quantity_loaded INTEGER NOT NULL CHECK (quantity_loaded > 0)
       - quantity_sold INTEGER NOT NULL DEFAULT 0 CHECK (quantity_sold >= 0)
       - quantity_returned INTEGER NOT NULL DEFAULT 0 CHECK (quantity_returned >= 0)
       - created_at TIMESTAMPTZ NOT NULL DEFAULT now()
       - UNIQUE(load_id, product_id) — one entry per product per load

    **Indexes:**
    - UNIQUE partial index on driver_loads(driver_id) WHERE status = 'active' — enforces one active load per driver
    - Index on driver_loads(business_id)
    - Index on driver_loads(driver_id, status)
    - Index on driver_load_items(load_id)

    **Triggers:**
    - set_updated_at trigger on driver_loads (reuse existing trigger function from migration 018)

    **RLS Policies (ENABLE ROW LEVEL SECURITY on both tables):**

    driver_loads:
    - SELECT: role IN ('owner', 'admin') see all for their business_id; role='driver' sees only own loads (driver_id = auth.uid())
    - INSERT: blocked via RLS — all inserts go through SECURITY DEFINER RPC
    - UPDATE: blocked — state changes through RPCs only
    - DELETE: blocked — no deletes

    driver_load_items: <!-- audit-added: explicit per-table RLS — Supabase RLS does NOT cascade through FKs -->
    - SELECT: use EXISTS subquery to driver_loads for business_id + role check:
      ```sql
      CREATE POLICY driver_load_items_owner_admin_select ON driver_load_items
        FOR SELECT USING (EXISTS (
          SELECT 1 FROM driver_loads dl
          WHERE dl.id = driver_load_items.load_id
          AND dl.business_id = get_user_business_id()
          AND get_user_role() IN ('owner', 'admin')
        ));
      CREATE POLICY driver_load_items_driver_select ON driver_load_items
        FOR SELECT USING (EXISTS (
          SELECT 1 FROM driver_loads dl
          WHERE dl.id = driver_load_items.load_id
          AND dl.driver_id = auth.uid()
          AND get_user_role() = 'driver'
        ));
      ```
    - INSERT/UPDATE/DELETE: blocked — managed by RPCs

    **RPC — create_driver_load(p_business_id UUID, p_driver_id UUID, p_items JSONB, p_notes TEXT DEFAULT NULL):**
    - SECURITY DEFINER, LANGUAGE plpgsql
    - Returns UUID (the new load ID)
    - Auth check: caller must be owner or admin (use get_user_role() pattern from existing RPCs — `IF get_user_role() NOT IN ('owner', 'admin') THEN RAISE EXCEPTION 'unauthorized: only owner/admin can create loads'; END IF;`)
    - Business check: p_business_id must match caller's business_id (`IF p_business_id != get_user_business_id() THEN RAISE EXCEPTION 'unauthorized: business_id mismatch'; END IF;`)
    - Driver validation: verify p_driver_id is a user with role='driver' in same business <!-- audit-added -->
      (`IF NOT EXISTS (SELECT 1 FROM users WHERE id = p_driver_id AND role = 'driver' AND business_id = p_business_id) THEN RAISE EXCEPTION 'invalid_driver: driver not found in business'; END IF;`)
    - Empty items guard: reject if p_items IS NULL or jsonb_array_length(p_items) = 0 <!-- audit-added -->
      (`IF p_items IS NULL OR jsonb_array_length(p_items) = 0 THEN RAISE EXCEPTION 'invalid_items: at least one product required'; END IF;`)
    - Guard: SELECT from driver_loads WHERE driver_id = p_driver_id AND status = 'active' — if exists, RAISE EXCEPTION 'driver_has_active_load'
    - For each item in p_items array:
      - Validate product exists and belongs to p_business_id
      - Check stock_on_hand >= requested quantity WITH FOR UPDATE row lock to prevent race conditions <!-- audit-added: matches create_order_atomic pattern -->
        (`SELECT stock_on_hand INTO v_stock FROM products WHERE id = v_product_id AND business_id = p_business_id FOR UPDATE;`)
      - If v_stock < requested quantity, RAISE EXCEPTION 'insufficient_stock' with product details
    - INSERT INTO driver_loads (business_id, driver_id, notes) → capture load_id
    - For each item: INSERT INTO driver_load_items (load_id, product_id, quantity_loaded)
    - For each item: UPDATE products SET stock_on_hand = stock_on_hand - quantity WHERE id = product_id
    - For each item: INSERT INTO stock_movements (business_id, product_id, movement_type, quantity, reference_id, notes, created_by)
      - movement_type = 'load_out', quantity = negative loaded amount, reference_id = load_id
    - RETURN load_id

    **RPC — get_driver_loads(p_business_id UUID):**
    - SECURITY DEFINER, returns TABLE
    - Returns all loads for business with driver name (join users), item count, total quantity, status
    - Auth check: caller must be owner/admin OR driver (driver sees only own)
    - Order by opened_at DESC

    **Grants:** <!-- audit-added: follow migration 018 REVOKE pattern -->
    - REVOKE EXECUTE on create_driver_load FROM PUBLIC;
    - REVOKE EXECUTE on get_driver_loads FROM PUBLIC;
    - GRANT EXECUTE on both RPCs to authenticated
    - REVOKE from anon

    **Deployment step:** Run migration against Supabase:
    ```
    supabase db push
    ```
    or apply via SQL editor.
