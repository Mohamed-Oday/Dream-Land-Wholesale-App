<objective>
## Goal
Complete the driver stock loading lifecycle: drivers see their loaded stock, close shifts with return receipt, orders track driver-level sales, and admin/owner can add items to active loads.

## Purpose
Plan 11-01 created the load-creation side. This plan closes the loop — drivers operate with their loaded stock throughout the day, orders track what they sell, and shift close reconciles loaded vs sold vs returned. Admin can also fix loads by adding items if the initial load was wrong.

## Output
- Migration 021: close_driver_load RPC, add_to_driver_load RPC, modify create_order_atomic
- Driver stock screen (shows active load with loaded/sold/remaining)
- Shift close screen + return receipt
- Add-to-load screen for admin/owner
- Driver shell updated with stock tab
</objective>

<acceptance_criteria>

## AC-1: Close Driver Load
```gherkin
Given a driver has an active load with items (some sold via orders)
When admin/owner or driver initiates shift close with remaining quantities
Then driver_load_items.quantity_returned is set for each item,
  returned stock is added back to products.stock_on_hand,
  stock_movements are created with movement_type='load_return',
  and driver_loads.status is set to 'closed' with closed_at timestamp
```

## AC-2: Order Tracks Driver Sales
```gherkin
Given a driver has an active load containing product X
When the driver creates an order that includes product X
Then driver_load_items.quantity_sold is incremented by the ordered quantity
  (in addition to existing warehouse stock deduction and stock_movements logging)
```

## AC-3: Driver Stock View
```gherkin
Given a driver has an active load
When the driver opens the stock tab in their shell
Then they see each product with columns: loaded | sold | remaining
  where remaining = quantity_loaded - quantity_sold
  and a "Close Shift" button is visible
```

## AC-4: Return Receipt
```gherkin
Given a driver load was just closed
When the user views the return receipt screen and taps Print
Then a Bluetooth receipt prints showing:
  driver name, each product with loaded/sold/returned quantities,
  and timestamp
```

## AC-5: Add to Active Load
```gherkin
Given a driver has an active load
When admin/owner adds products to the load
Then new items are added to driver_load_items (or quantities increased),
  warehouse stock is deducted,
  stock_movements are logged with 'load_out',
  and the load receipt updates to show all items
```

    Use CREATE OR REPLACE FUNCTION to redefine create_order_atomic with the SAME signature and existing logic, plus a new section after Step 5:
    ```sql
    -- ── Step 6: Track driver load sales (if driver has active load) ──
    SELECT id INTO v_active_load_id
    FROM driver_loads
    WHERE driver_id = v_driver_id AND status = 'active';

    IF v_active_load_id IS NOT NULL THEN
      FOR v_line IN SELECT * FROM jsonb_array_elements(p_line_items)
      LOOP
        UPDATE driver_load_items
        SET quantity_sold = quantity_sold + (v_line->>'quantity')::INTEGER
        WHERE load_id = v_active_load_id
          AND product_id = (v_line->>'product_id')::UUID;
        -- No error if product not in load (driver might sell warehouse-only items)
      END LOOP;
    END IF;
    ```
    IMPORTANT: Declare v_active_load_id UUID in the DECLARE block. Keep ALL existing logic identical — only add Step 6 before the RETURN statement.

    **Modify cancel_order — Reverse driver_load_items.quantity_sold on cancellation:** <!-- audit-added: MUST-HAVE -->
    Use CREATE OR REPLACE FUNCTION to redefine cancel_order (from migration 018) with ALL existing logic, plus after the store credit_balance reversal add:
    ```sql
    -- ── Reverse driver load sales (if driver had active load) ──
    SELECT id INTO v_active_load_id
    FROM driver_loads
    WHERE driver_id = v_order.driver_id AND status = 'active';

    IF v_active_load_id IS NOT NULL THEN
      FOR v_line IN
        SELECT product_id, quantity FROM order_lines WHERE order_id = p_order_id
      LOOP
        UPDATE driver_load_items
        SET quantity_sold = GREATEST(quantity_sold - v_line.quantity, 0)
        WHERE load_id = v_active_load_id
          AND product_id = v_line.product_id;
      END LOOP;
    END IF;

    -- ── Restore warehouse stock (fixes pre-existing gap) ──
    FOR v_line IN
      SELECT product_id, quantity FROM order_lines WHERE order_id = p_order_id
    LOOP
      UPDATE products SET stock_on_hand = stock_on_hand + v_line.quantity
      WHERE id = v_line.product_id;

      INSERT INTO stock_movements (business_id, product_id, movement_type, quantity, reference_id, created_by)
      VALUES (v_order.business_id, v_line.product_id, 'cancellation_restore', v_line.quantity, p_order_id, auth.uid());
    END LOOP;
    ```
    IMPORTANT: Read the FULL cancel_order from migration 018 first. Add v_active_load_id UUID and v_line RECORD to DECLARE. Keep existing REVOKE/GRANT signature.
