<objective>
## Goal
Consolidate 5 separate dashboard queries into a single PostgreSQL RPC and document the role-operation matrix — the final plan of milestone v0.2.1.

## Purpose
**Dashboard consolidation:** The owner dashboard makes 5+ separate Supabase queries on every load (revenue, order count, purchases, top debtors, low stock). On mobile networks, this creates visible latency and 5 separate loading states. A single RPC returns all KPI data in one round trip.

**Role-operation matrix:** As the codebase has grown to 19 migrations and dozens of RLS policies, there's no single document mapping which roles can perform which operations. This is a basic compliance artifact for access control documentation.

## Output
- Migration `019_dashboard_rpc.sql` with `get_dashboard_summary()` function
- Refactored dashboard repository + providers using single RPC
- Role-operation matrix documentation
</objective>

<task type="auto">
  <name>Task 1: Migration 019 — Consolidated Dashboard RPC</name>
  <files>supabase/migrations/019_dashboard_rpc.sql</files>
  <action>
    Create `get_dashboard_summary(p_business_id UUID)` RETURNS JSONB:

    **Auth:**
    - `auth.uid() IS NOT NULL`
    - `get_user_role() IN ('owner', 'admin')` — only owner/admin see dashboard
    - `p_business_id = get_user_business_id()`

    **Body — compute in SQL, return as JSONB:**
    - <!-- audit-added --> Use COALESCE on ALL aggregates — SUM() and jsonb_agg() return NULL on empty sets, which will crash Dart `as double` casts
    - `today_revenue`: COALESCE(SUM(amount), 0) from payments WHERE business_id AND created_at >= today start
    - `today_order_count`: COUNT(*) from orders WHERE business_id AND created_at >= today start
    - `today_purchases`: COALESCE(SUM(total_cost), 0) from purchase_orders WHERE business_id AND created_at >= today start
    - `today_profit`: today_revenue - today_purchases (computed from COALESCE'd values)
    - `top_debtors`: COALESCE(jsonb_agg(...), '[]'::jsonb) — SELECT id, name, credit_balance FROM stores WHERE credit_balance > 0 ORDER BY credit_balance DESC LIMIT 5
    - `low_stock_products`: COALESCE(jsonb_agg(...), '[]'::jsonb) — SELECT id, name, stock_on_hand, low_stock_threshold FROM products WHERE active AND low_stock_threshold > 0 AND stock_on_hand <= low_stock_threshold

    **Algeria timezone:** Use `(now() AT TIME ZONE 'Africa/Algiers')::date` for today's start boundary, converting to UTC for filtering. This is more robust than the Dart-side UTC+1 offset calculation.

    **Return format:**
    ```json
    {
      "today_revenue": 12500.00,
      "today_order_count": 15,
      "today_purchases": 3000.00,
      "today_profit": 9500.00,
      "top_debtors": [{"id": "...", "name": "...", "credit_balance": 500}],
      "low_stock_products": [{"id": "...", "name": "...", "stock_on_hand": 2, "low_stock_threshold": 5}]
    }
    ```

    - SECURITY DEFINER + REVOKE FROM PUBLIC + GRANT TO authenticated
    - Header comment: "Deploy via supabase db push or Supabase Dashboard SQL Editor"

    Avoid:
    - Do NOT include recent_orders (has complex PostgREST joins — keep as separate query)
    - Do NOT include package_alerts (already an RPC — keep as-is)
    - Do NOT include pending_discounts (from order repo, different refresh lifecycle)
    - Do NOT drop existing individual functions — they may be used elsewhere
  </action>
  <verify>
    1. SQL syntax review — function returns valid JSONB
    2. Algeria timezone handled in SQL (not Dart-side offset)
    3. Role check restricts to owner/admin
    4. REVOKE/GRANT pattern matches established conventions
  </verify>
  <done>AC-1 (single RPC returns all KPI data) satisfied at database level</done>
