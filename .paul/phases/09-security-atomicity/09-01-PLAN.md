<objective>
## Goal
Harden the authorization layer: prevent JWT metadata tampering, add role checks to all SECURITY DEFINER functions, and make the balance_adjustments audit table append-only with role-based access.

## Purpose
F-04-001 (JWT metadata spoofable) is the single most critical finding — it enables complete privilege escalation. F-02-007/F-04-007 (missing role checks) allows any authenticated user to invoke owner-only operations. F-05-005 (audit trail writable by drivers) undermines the core accountability guarantee. All three are SQL-only fixes in one migration.

## Output
- New SQL migration `016_security_hardening.sql` with 3 security fixes
- All SECURITY DEFINER functions protected by role checks
- Audit trail made append-only
</objective>

  <name>Task 1: Create JWT metadata protection trigger</name>
  <files>supabase/migrations/016_security_hardening.sql</files>
  <action>
    Create migration file starting with a trigger on `auth.users` that prevents modification of `role` and `business_id` in `raw_user_meta_data` after initial creation:

    ```sql
    -- Trigger: prevent client-side modification of role and business_id
    -- auth.updateUser() can modify user_metadata, but role/business_id must be immutable
    CREATE OR REPLACE FUNCTION protect_user_metadata()
    RETURNS TRIGGER AS $$
    BEGIN
      -- On UPDATE: if role or business_id changed, revert to OLD values
      IF OLD.raw_user_meta_data IS NOT NULL THEN
        -- Preserve role from being changed
        IF (OLD.raw_user_meta_data->>'role') IS NOT NULL
           AND (NEW.raw_user_meta_data->>'role') IS DISTINCT FROM (OLD.raw_user_meta_data->>'role') THEN
          NEW.raw_user_meta_data := jsonb_set(
            NEW.raw_user_meta_data,
            '{role}',
            to_jsonb(OLD.raw_user_meta_data->>'role')
          );
        END IF;
        -- Preserve business_id from being changed
        IF (OLD.raw_user_meta_data->>'business_id') IS NOT NULL
           AND (NEW.raw_user_meta_data->>'business_id') IS DISTINCT FROM (OLD.raw_user_meta_data->>'business_id') THEN
          NEW.raw_user_meta_data := jsonb_set(
            NEW.raw_user_meta_data,
            '{business_id}',
            to_jsonb(OLD.raw_user_meta_data->>'business_id')
          );
        END IF;
      END IF;
      RETURN NEW;
    END;
    $$ LANGUAGE plpgsql SECURITY DEFINER;

    CREATE TRIGGER protect_user_metadata_trigger
      BEFORE UPDATE ON auth.users
      FOR EACH ROW
      EXECUTE FUNCTION protect_user_metadata();
    ```

    Key design decisions:
    - Uses BEFORE UPDATE trigger (not RAISE EXCEPTION) so the updateUser call succeeds but the protected fields silently revert. This prevents app crashes if other metadata fields are legitimately updated.
    - Only protects `role` and `business_id`. Other metadata fields can still be changed.
    - Allows initial INSERT (signUp) to set these fields freely.
    - Allows changes when old value is NULL (first-time setup).

    Avoid: Using RAISE EXCEPTION which would break any auth.updateUser() call that touches any metadata field.
  </action>
  <verify>
    - SQL syntax is valid (CREATE FUNCTION + CREATE TRIGGER)
    - Trigger is BEFORE UPDATE (not AFTER)
    - Trigger is on auth.users (not public.users)
    - Logic preserves OLD values, doesn't raise exception
    - INSERT (signUp) is not affected (trigger is UPDATE only)
  </verify>
  <done>AC-1 satisfied: JWT metadata role/business_id protected from client modification</done>

<task type="auto">
  <name>Task 3: Fix balance_adjustments RLS to append-only with role access</name>
  <files>supabase/migrations/016_security_hardening.sql</files>
  <action>
    Append to the same migration file:

    ```sql
    -- Drop the overly permissive FOR ALL policy
    DROP POLICY IF EXISTS "Users can manage own business adjustments" ON balance_adjustments;

    -- Owner: SELECT + INSERT (via RPC). No direct UPDATE/DELETE.
    CREATE POLICY "balance_adjustments_owner_read"
      ON balance_adjustments FOR SELECT
      USING (get_user_role() = 'owner' AND business_id = get_user_business_id());

    CREATE POLICY "balance_adjustments_owner_insert"
      ON balance_adjustments FOR INSERT
      WITH CHECK (get_user_role() = 'owner' AND business_id = get_user_business_id());

    -- Admin: SELECT only
    CREATE POLICY "balance_adjustments_admin_read"
      ON balance_adjustments FOR SELECT
      USING (get_user_role() = 'admin' AND business_id = get_user_business_id());

    -- Driver: SELECT only (can see adjustments for their business)
    CREATE POLICY "balance_adjustments_driver_read"
      ON balance_adjustments FOR SELECT
      USING (get_user_role() = 'driver' AND business_id = get_user_business_id());

    -- No UPDATE or DELETE policies for any role = append-only
    ```

    Note: The adjust_store_balance RPC runs as SECURITY DEFINER (bypasses RLS), so the owner INSERT policy is for any future direct inserts. The RPC is the primary entry point.
