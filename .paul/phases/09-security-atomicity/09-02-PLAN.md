<objective>
## Goal
Consolidate the 5-step order creation into a single atomic PostgreSQL RPC, enforce deactivation via RLS, and add a blocking startup version check — the three remaining security/integrity fixes in Phase 9.

## Purpose
**Atomic order:** Currently, order creation spans 5 separate database operations (INSERT order, INSERT lines, UPDATE store balance, CREATE package logs, DEDUCT stock). If any step fails after the order is saved, the database is left in an inconsistent state — store balance wrong, packages untracked, stock not deducted. This collapses ~10 AEGIS findings into a single transactional guarantee.

**Deactivation enforcement:** Setting `active = false` on a user currently does nothing at the database level — their JWT still passes all RLS checks. A deactivated driver can continue creating orders.

**Version check:** No mechanism exists to force users onto a minimum version. If a breaking migration ships, old APKs will silently fail against the new schema.

## Output
- Migration `017_atomic_order_and_enforcement.sql` (atomic RPC + deactivation enforcement)
- Refactored `OrderRepository.create()` calling single RPC
- Simplified `CreateOrderScreen._confirmAndSubmit()` with no fire-and-forget calls
- Blocking force-update screen on startup when `min_version` > current
</objective>

<acceptance_criteria>

## AC-1: Atomic Order Creation
```gherkin
Given a driver with a valid cart (store selected, products with stock, optional discount)
When the driver submits the order
Then a single RPC call atomically:
  - Creates the order record with status 'created'
  - Inserts all order_lines
  - Increases the store's credit_balance by the order total
  - Creates package_logs for all returnable products
  - Deducts stock_on_hand for each product and logs stock_movements
And the Dart code makes exactly ONE network call (no fire-and-forget side calls)
```

## AC-2: Atomic Rollback on Failure
```gherkin
Given a driver submits an order
When any step inside the atomic RPC fails (e.g., FK violation, stock negative)
Then NO partial state is persisted — no order, no lines, no balance change, no logs
And the Dart code shows an error message to the driver
```

## AC-3: Deactivation Blocks All Access
```gherkin
Given an owner sets a user's active = false in the users table
When the deactivated user attempts any data operation (SELECT, INSERT, UPDATE)
Then all RLS policies deny access because get_user_role() returns NULL for inactive users
And the user effectively cannot read or write any business data
```

## AC-4: Startup Version Check — Blocking
```gherkin
Given the app starts and remote_config min_version is greater than AppConstants.appVersion
When the router initializes
Then a blocking force-update screen is shown
And the user cannot navigate to any app feature (login, driver shell, etc.)
And the screen displays the min_version and a download prompt
```

## AC-5: Idempotent Retry on Network Timeout <!-- audit-added -->
```gherkin
Given a driver submits an order with a client-generated order UUID
When the network times out and the client retries the same RPC call with the same UUID
Then the RPC detects the order already exists and returns the existing order data
And no duplicate order, lines, balance change, or stock deduction occurs
```

## AC-6: Cross-Business Store Protection <!-- audit-added -->
```gherkin
Given a driver attempts to create an order for a store belonging to a different business
When the atomic RPC validates the store
Then the RPC raises an authorization exception
And no order is created
```

## AC-7: Startup Version Check — Pass-Through
```gherkin
Given the app starts and remote_config min_version is less than or equal to AppConstants.appVersion
When the router initializes
Then the app proceeds normally to the auth flow
And the existing settings-screen update notification continues to work independently
```

  <action>
    Create migration with three parts:

    **PART 1: `create_order_atomic()` function**
    - Parameters: `p_order_id UUID DEFAULT NULL, p_store_id UUID, p_business_id UUID, p_subtotal NUMERIC, p_tax_percentage NUMERIC, p_tax_amount NUMERIC, p_discount NUMERIC, p_discount_status TEXT, p_total NUMERIC, p_line_items JSONB`
    - `p_line_items` is a JSON array of `{product_id, quantity, unit_price, line_total}`
    - Auth: `auth.uid() IS NOT NULL`, `get_user_role() = 'driver'`, `p_business_id = get_user_business_id()`
    - <!-- audit-added --> Idempotency: If `p_order_id` is provided, check `SELECT 1 FROM orders WHERE id = p_order_id` — if exists, return the existing order data (idempotent retry on network timeout). If NULL, generate via `gen_random_uuid()`
    - <!-- audit-added --> Store-business validation: `SELECT 1 FROM stores WHERE id = p_store_id AND business_id = p_business_id` — RAISE EXCEPTION 'unauthorized: store not in business' if not found. Prevents cross-business writes since SECURITY DEFINER bypasses RLS.
    - <!-- audit-added --> Discount status validation: `IF p_discount_status NOT IN ('none', 'pending') THEN RAISE EXCEPTION 'invalid discount_status'` — only 'none' and 'pending' are valid at order creation time
    - Step 1: Use `p_order_id` if provided, else generate order UUID via `gen_random_uuid()`
    - Step 2: INSERT into orders with `driver_id = auth.uid()` (server-derived, not client parameter)
    - Step 3: Loop `jsonb_array_elements(p_line_items)` → INSERT order_lines with generated UUIDs
    - Step 4: `UPDATE stores SET credit_balance = credit_balance + p_total WHERE id = p_store_id`
    - Step 5: Loop line items → JOIN products to check `has_returnable_packaging` → if true, insert package_log entry with balance calculation (reuse create_package_log logic: SELECT latest balance_after FOR UPDATE, compute new balance, INSERT log)
    - Step 6: Loop line items → `UPDATE products SET stock_on_hand = stock_on_hand - quantity` + INSERT stock_movements
    - Return: JSONB object with `{id, store_id, driver_id, business_id, subtotal, tax_percentage, tax_amount, discount, discount_status, total, status, created_at}`
    - SECURITY DEFINER with `REVOKE EXECUTE FROM PUBLIC` + `GRANT EXECUTE TO authenticated`

    **PART 2: Deactivation enforcement on `get_user_role()`**
    - `CREATE OR REPLACE FUNCTION get_user_role()` — overwrite existing
    - Add: `SELECT active INTO v_active FROM users WHERE id = auth.uid()`
    - If `v_active = false` → `RETURN NULL` (blocks ALL RLS policies system-wide)
    - If `v_active IS NULL` (no row — edge case during init) → fall through to JWT role (preserve backward compat)
    - If `v_active = true` → return JWT role as before: `auth.jwt() -> 'user_metadata' ->> 'role'`

    **PART 3: Deployment step comment**
    - Add header comment: "Deploy via supabase db push or Supabase Dashboard SQL Editor"

    Avoid:
    - Do NOT drop or modify `update_store_balance_on_order`, `create_package_log`, or `deduct_stock_for_order` — they are still used by other flows (cancellation, standalone package collection, purchase replenishment)
    - Do NOT add CHECK constraints on stock_on_hand (that's Phase 10 scope)
    - Do NOT use `RAISE EXCEPTION` for the deactivation check in `get_user_role` — just return NULL silently
  </action>
