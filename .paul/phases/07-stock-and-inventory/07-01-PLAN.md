<objective>
## Goal
Add stock tracking columns to products, create a stock_movements audit table, and wire automatic stock deduction on order creation plus stock replenishment on purchase order creation — so product stock levels stay accurate without manual intervention.

## Purpose
The owner needs to know current inventory levels at a glance. Without stock tracking, there's no way to know if a product is running low until it's physically gone — causing missed sales and emergency supplier runs. This is the data foundation for low stock alerts and manual adjustments (Plan 07-02).

## Output
- Migration 013: `stock_on_hand` + `low_stock_threshold` columns on products, `stock_movements` table, 3 RPC functions
- Product list shows stock count per product
- Orders automatically deduct stock on creation
- Purchase orders automatically replenish stock on creation
- Order cancellation restores deducted stock
- All stock changes logged in stock_movements for audit trail
</objective>

    **2. Create stock_movements table:**
    - id UUID PRIMARY KEY DEFAULT gen_random_uuid()
    - business_id UUID NOT NULL (FK to users — follows existing pattern, NO FK constraint — matches products/orders pattern)
    - product_id UUID NOT NULL REFERENCES products(id)
    - movement_type TEXT NOT NULL CHECK (movement_type IN ('order_out', 'purchase_in', 'cancellation_restore', 'adjustment'))
    - quantity INTEGER NOT NULL (negative for deductions, positive for additions)
    - reference_id UUID (order_id or purchase_order_id — nullable for adjustments)
    - notes TEXT DEFAULT ''
    - created_by UUID NOT NULL REFERENCES users(id)
    - created_at TIMESTAMPTZ NOT NULL DEFAULT now()
    - Index on (product_id, created_at DESC) for movement history queries
    - Index on (business_id) for RLS performance

    **3. RLS policies on stock_movements:**
    - Follow existing RLS pattern from 001_initial_schema.sql
    - Owner: full SELECT, INSERT within business_id
    - Admin: SELECT, INSERT within business_id
    - <!-- audit-added --> Driver: SELECT only within business_id (drivers affect stock via SECURITY DEFINER RPCs, NOT direct INSERT — direct INSERT would allow stock manipulation outside order flow)
    - All policies use get_user_role() from user_metadata (established pattern)

    **4. RPC function: deduct_stock_for_order(p_order_id UUID)**
    - Verify auth.uid() is not null (security check — follows 006/007 pattern)
    - <!-- audit-added --> Verify the order's business_id matches caller's JWT business_id: (auth.jwt() -> 'user_metadata' ->> 'business_id')::UUID — prevents cross-business stock manipulation
    - <!-- audit-added --> Idempotency guard: IF EXISTS (SELECT 1 FROM stock_movements WHERE reference_id = p_order_id AND movement_type = 'order_out') THEN RETURN — prevents double-deduction on network retry
    - For each order_line of the order: UPDATE products SET stock_on_hand = stock_on_hand - quantity
    - Insert stock_movement for each line (movement_type = 'order_out', quantity as negative, reference_id = order_id, created_by = driver_id from order)
    - SECURITY DEFINER (follows existing RPC pattern)
    - GRANT EXECUTE to authenticated

    **5. RPC function: replenish_stock_from_purchase(p_purchase_order_id UUID)**
    - Verify auth.uid() is not null
    - <!-- audit-added --> Verify the PO's business_id matches caller's JWT business_id — same pattern as deduct
    - <!-- audit-added --> Idempotency guard: IF EXISTS (SELECT 1 FROM stock_movements WHERE reference_id = p_purchase_order_id AND movement_type = 'purchase_in') THEN RETURN — prevents double-replenishment
    - For each purchase_order_line: UPDATE products SET stock_on_hand = stock_on_hand + quantity
    - Insert stock_movement for each line (movement_type = 'purchase_in', quantity as positive, reference_id = purchase_order_id, created_by from PO)
    - SECURITY DEFINER + GRANT EXECUTE to authenticated

    **6. RPC function: restore_stock_for_cancellation(p_order_id UUID)**
    - Verify auth.uid() is not null
    - <!-- audit-added --> Verify the order's business_id matches caller's JWT business_id — same pattern as deduct
    - <!-- audit-added --> Idempotency guard: IF EXISTS (SELECT 1 FROM stock_movements WHERE reference_id = p_order_id AND movement_type = 'cancellation_restore') THEN RETURN — prevents double-restoration
    - For each order_line of the order: UPDATE products SET stock_on_hand = stock_on_hand + quantity
    - Insert stock_movement for each line (movement_type = 'cancellation_restore', quantity as positive, reference_id = order_id, created_by = auth.uid())
    - SECURITY DEFINER + GRANT EXECUTE to authenticated

    Avoid: Creating triggers (explicit RPC calls are easier to debug and match existing patterns)
    Avoid: Adding FOREIGN KEY on business_id (matches existing pattern — no FK to users for business_id)
