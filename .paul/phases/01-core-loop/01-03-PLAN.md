<objective>
## Goal
Implement CRUD operations for Products and Stores — the two core entities that Orders depend on. Owner/Admin can create, edit, and deactivate products. Admin/Driver can create and edit stores. All data syncs to Supabase with Drift local caching.

## Purpose
Without products, there's nothing to order. Without stores, there's nowhere to deliver. These are the foundation entities for the order flow in Plan 01-04. This plan also establishes the data access pattern (Repository → Provider → UI) used by all future features.

## Output
- Product list + add/edit screens (Owner/Admin only)
- Store list + add/edit screens (all roles can view, Admin/Driver can create)
- Repository pattern for Supabase CRUD with Drift local cache
- Riverpod providers for reactive data
</objective>

  <action>
    1. Create ProductRepository (product_repository.dart):
       - Constructor takes SupabaseClient and AppUser (for business_id)
       - getAll(): fetch from Supabase 'products' table filtered by business_id, where active=true
       - getById(id): fetch single product
       - create(name, unitPrice, unitsPerPackage, hasReturnablePackaging): insert to Supabase with business_id from current user
       - update(id, fields): update in Supabase
       - deactivate(id): set active=false (soft delete)
       - All methods return typed data (Map or simple model class)

    2. Create StoreRepository (store_repository.dart):
       - Constructor takes SupabaseClient and AppUser
       - getAll(): fetch from Supabase 'stores' table filtered by business_id
       - getById(id): fetch single store
       - create(name, address, phone, contactPerson): insert to Supabase with business_id
       - update(id, fields): update in Supabase
       - All methods return typed data

    3. Create ProductProvider (product_provider.dart):
       - productRepositoryProvider: Provider that creates ProductRepository with current user's business_id
       - productListProvider: FutureProvider that calls repository.getAll()
       - Methods to invalidate/refresh the list after create/edit

    4. Create StoreProvider (store_provider.dart):
       - storeRepositoryProvider: Provider
       - storeListProvider: FutureProvider that calls repository.getAll()
       - Methods to invalidate/refresh after create/edit

    Pattern: Keep repositories simple — direct Supabase calls, no Drift caching yet (caching comes when offline-first is refined in Phase 4). Providers handle reactivity.

    Avoid: Do NOT implement complex offline-first sync in this plan — direct Supabase CRUD is sufficient for MVP.
    Avoid: Do NOT add pagination — 10-20 stores and <50 products don't need it.
