<objective>
## Goal
Implement username/password authentication with role-based routing. After login, the app routes to the correct navigation shell based on user role (owner → /owner, admin → /admin, driver → /driver). Logout returns to login screen.

## Purpose
Without auth, the app has no access control and defaults to a hardcoded route. Auth is the gate that enables all role-specific features in Plans 01-03 and 01-04. The owner, admin, and driver experiences diverge entirely after login.

## Output
- Login screen with username/password form (Arabic RTL)
- Auth service that authenticates against Supabase Auth
- Riverpod auth state provider that persists session
- GoRouter redirect logic that routes by role
- Logout functionality
- Seed SQL for initial owner account
</objective>

       - No password field — only used during login

    2. Create AuthService (auth_service.dart):
       - Uses Supabase Auth (not custom JWT)
       - Auth approach: Use Supabase Auth email field as "username@tawzii.local" format
         (Supabase requires email format, so we append a dummy domain)
       - IMPORTANT: Supabase project must have email confirmation DISABLED <!-- audit-added -->
         (Settings → Auth → Email → "Enable email confirmations" OFF)
         Document this in seed SQL comments
       - signIn(username, password): calls Supabase.auth.signInWithPassword(email: '$username@tawzii.local', password: password)
       - Client-side login throttle: track failed attempts, disable login for 30s after 3 consecutive failures <!-- audit-added -->
       - signOut(): calls Supabase.auth.signOut()
       - getCurrentUser(): returns AppUser from current session or null
       - getRole(): extracts role from user_metadata (set during account creation)
       - onAuthStateChange: stream of auth state changes

    3. Create AuthProvider (auth_provider.dart) using Riverpod:
       - authStateProvider: StreamProvider that listens to Supabase auth state changes
       - currentUserProvider: Provider<AppUser?> derived from auth state
       - isLoggedInProvider: Provider<bool>
       - userRoleProvider: Provider<String?> (owner, admin, driver)
       - loginProvider: FutureProvider.family for login action
       - logoutProvider: Provider for logout action

    4. Create seed SQL (002_seed_owner.sql):
       - Document that owner account must be created via Supabase Auth dashboard or Edge Function
       - IMPORTANT: When creating auth user, set user_metadata with role and business_id: <!-- audit-added -->
         {"role": "owner", "business_id": "[uuid]", "name": "المالك", "username": "owner"}
         This metadata is what AppUser.fromSupabaseUser reads
       - Document: Supabase Dashboard → Authentication → Create User → set email to "owner@tawzii.local", set raw_user_meta_data
       - Provide SQL to insert into public.users table after auth user is created:
         INSERT INTO users (id, business_id, name, username, role)
         VALUES ('[auth-user-uuid]', '[business-uuid]', 'المالك', 'owner', 'owner');
       - Include comment explaining the full setup process (disable email confirmation, create auth user with metadata, insert public.users row)

    Avoid: Do NOT store passwords locally in Drift — auth is Supabase-only.
    Avoid: Do NOT implement user creation flow in this plan — that's admin functionality in Plan 01-03 or Phase 3.
