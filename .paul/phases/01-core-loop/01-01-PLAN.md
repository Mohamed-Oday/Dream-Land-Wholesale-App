<objective>
## Goal
Set up the complete project foundation: Flutter app scaffold with RTL Arabic support and Dream Land Shopping brand theme, Supabase database schema with RLS policies, and Drift local database with sync queue infrastructure.

## Purpose
This is the foundation that every other plan in Phase 1 builds on. Without the Flutter scaffold, there's no app to add screens to. Without the Supabase schema, there's no backend. Without Drift, there's no offline-first capability. All three must exist before auth, CRUD, or order flows can be implemented.

## Output
- A runnable Flutter app with orange Material 3 theme, RTL Arabic layout, and bottom navigation shells for each role
- Supabase migration SQL with all tables (User, Store, Product, Order, OrderLine, Payment, PackageLog, DriverLocation, AppConfig) and RLS policies
- Drift database with matching local tables and a sync queue mechanism
</objective>

       - Drivers: SELECT/INSERT own rows (where driver_id = auth.uid() or created_by = auth.uid())

       - Include all columns from the Supabase schema
       - Add business_id to all tables

    2. Create sync_queue_table.dart:
       - id (integer, autoIncrement, PK)
       - table_name (text) — which Supabase table
       - operation (text) — insert|update|delete
       - record_id (text) — UUID of the record
       - payload (text) — JSON-serialized data
       - created_at (dateTime)
       - synced (boolean, default false)
       - retry_count (integer, default 0)
       - error_message (text, nullable)

    3. Create app_database.dart:
       - @DriftDatabase annotation including all tables
       - Database version and migration strategy
       - Lazy singleton pattern for database instance

    4. Create sync_queue.dart:
       - Methods: enqueue(tableName, operation, recordId, payload)
       - Methods: getPending() — returns unsynced items ordered by created_at
       - Methods: markSynced(id), markFailed(id, errorMessage)

    5. Create sync_service.dart:
       - processQueue() — iterate pending items, push to Supabase, mark synced
       - Conflict resolution: server timestamp wins (on conflict, fetch server version)
       - Error handling: increment retry_count, store error_message, skip after 3 retries

    6. Create supabase_client.dart:
       - Wrapper around SupabaseFlutter initialization
       - Helper methods for common operations (typed responses)

    7. Run build_runner to generate Drift code:
       dart run build_runner build --delete-conflicting-outputs
