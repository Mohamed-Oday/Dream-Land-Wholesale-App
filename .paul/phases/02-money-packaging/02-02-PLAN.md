<objective>
## Goal
Driver can track returnable packaging per-product per-store. Packages are auto-given when orders with returnable products are delivered. Driver can visit a store standalone to collect returned packages. Package balance is tracked per product per store.

## Purpose
Returnable packaging (crates/trays) is a significant cost for the business. Without tracking, packages get lost and the owner has no visibility into which stores hold how many. This plan makes package flow visible and accountable.

## Output
- `lib/features/packages/` — repository, providers, and 2 screens
- New Supabase migration with RPC function for atomic package balance tracking
- Modified order creation to auto-log packages given for returnable products
- Modified driver shell (Packages tab wired)
- Updated l10n with package-related Arabic strings
</objective>

## Supabase Schema Reference
package_logs: id(uuid), business_id(uuid), store_id(uuid), driver_id(uuid),
  product_id(uuid), order_id(uuid nullable), given(int >=0), collected(int >=0),
  balance_after(int), created_at(timestamptz)

products: has_returnable_packaging(bool), units_per_package(int nullable)

Business logic:
  - given = packages given to store (on order delivery)
  - collected = packages returned by store (on collection visit)
  - balance_after = running balance (positive = store holds our packages)
  - order_id = NULL for standalone collections, set for order-linked entries

## AC-1: Packages Auto-Logged on Order Creation
```gherkin
Given a driver creates an order with returnable products
When the order is saved successfully
Then a package_log entry is created for each returnable product
  with given = quantity (packages), collected = 0, and order_id set
  and balance_after reflects the updated running balance
```

## AC-2: Standalone Package Collection
```gherkin
Given a driver visits a store to collect returned packages
When they select the store, choose returnable products, enter collected amounts, and confirm
Then package_log entries are created with given = 0, collected = N, order_id = NULL
  and balance_after decreases by the collected amount
```

## AC-3: Package List Shows History
```gherkin
Given a driver has package activity (given and collected)
When they navigate to the Packages tab
Then they see a reverse-chronological list of package logs
  showing product name, store name, given/collected amounts, and balance_after
```

## AC-4: Empty Package State
```gherkin
Given a driver has no package logs
When they navigate to the Packages tab
Then they see an empty state with a message and FAB to start a collection
```

## AC-5: Package Collection Shows Current Balances
```gherkin
Given a store holds packages for multiple returnable products
When the driver opens the collection screen and selects that store
Then each returnable product shows its current package balance at that store
  and the driver can enter collected amounts per product
```

## AC-6: Collection Validation
```gherkin

Then the form shows validation errors and does not submit
```

    **Supabase Migration** (NEW file: supabase/migrations/003_package_functions.sql):

    1. `create_package_log(p_store_id UUID, p_product_id UUID, p_business_id UUID, p_given INT DEFAULT 0, p_collected INT DEFAULT 0, p_order_id UUID DEFAULT NULL)` → RETURNS JSONB:
       - Authorization: verify p_business_id matches caller's JWT business_id
       - <!-- audit-added --> **Row lock:** `SELECT balance_after INTO v_previous FROM package_logs WHERE store_id = p_store_id AND product_id = p_product_id ORDER BY created_at DESC LIMIT 1 FOR UPDATE` to prevent race conditions on concurrent package operations
       - If NOT FOUND, v_previous := 0
       - v_balance := v_previous + p_given - p_collected
       - INSERT into package_logs with all fields + auth.uid() as driver_id
       - RETURN jsonb with log id, previous_balance, balance_after

    2. `get_package_balances_for_store(p_store_id UUID)` → RETURNS TABLE (product_id UUID, balance INTEGER):
       - <!-- audit-added --> Returns current balance for ALL products at a store using DISTINCT ON:
       - `SELECT DISTINCT ON (product_id) product_id, balance_after AS balance FROM package_logs WHERE store_id = p_store_id ORDER BY product_id, created_at DESC`
       - More efficient than N individual queries
       - SECURITY DEFINER to allow driver access

    Add GRANT EXECUTE to anon and authenticated for both functions.

    **IMPORTANT:** Remind user to run migration in Supabase SQL Editor.

    **PackageRepository** (follow PaymentRepository pattern):
    - Constructor takes SupabaseClient + businessId
    - `getAll({String? driverId})` — select package_logs with product + store + driver joins
      - `.select('*, products(name), stores(name), users!package_logs_driver_id_fkey(name)')`
      - Optional driverId filter, ordered by created_at desc
    - `create({required String storeId, required String productId, int given = 0, int collected = 0, String? orderId})`:
      - Call RPC: `_client.rpc('create_package_log', params: {...})`
    - `getBalancesByStore(String storeId)` — get current balance for ALL returnable products at a store:
      - <!-- audit-added --> Call RPC: `_client.rpc('get_package_balances_for_store', params: {'p_store_id': storeId})`
      - Returns list of {product_id, balance} maps

    **PackageProvider**:

    - `packageListProvider` — FutureProvider for driver's package logs
    - `allPackageLogsProvider` — FutureProvider for owner (all logs)
