<objective>
## Goal
Set up end-to-end push notification infrastructure: Firebase Cloud Messaging (FCM) on the Flutter client, FCM token storage in Supabase, a Supabase Edge Function to send notifications via FCM v1 API, and foreground/background notification display.

## Purpose
Owner and admin need real-time alerts on their phones — even when the app is closed — for orders, payments, discounts, and stock events. This plan builds the plumbing; Plan 02 hooks it into all event triggers.

## Output
- Migration 022: `fcm_tokens` table with RLS + RPCs
- Supabase Edge Function: `send-notification` (FCM v1 API)
- Flutter: Firebase + FCM initialization, token lifecycle, notification display
- End-to-end: login → token stored → Edge Function callable → notification received
</objective>

## AC-1: FCM Token Registration
```gherkin
Given a user logs in on an Android device
When Firebase Messaging initializes and obtains an FCM token
Then the token is stored in the fcm_tokens table via upsert_fcm_token RPC
And the token is associated with the user's auth.uid()
```

## AC-2: Token Cleanup on Logout
```gherkin
Given a user has a stored FCM token
When the user logs out
Then the FCM token is deleted from fcm_tokens via delete_fcm_token RPC
```

## AC-3: Token Refresh
```gherkin
Given a user is logged in and the FCM token rotates
When onTokenRefresh fires
Then the new token is upserted in fcm_tokens (old entry updated or new entry added)
```

## AC-4: Edge Function Sends Notification
```gherkin
Given a user has a valid FCM token stored in fcm_tokens
When the send-notification Edge Function is invoked with event_type and data
Then FCM v1 API is called with the correct device token
And the device receives a push notification with title and body
```

## AC-5: Background Notification Display
```gherkin
Given the app is in the background or terminated
When a notification is received via FCM
Then the system notification tray displays the notification with title and body
```

## AC-6: Foreground Notification Display
```gherkin
Given the app is in the foreground
When a notification is received via FCM
Then a local notification is displayed matching system tray appearance
```

<task type="auto">
  <name>Task 1: Migration 022 — FCM tokens table + RPCs</name>
  <files>supabase/migrations/022_fcm_tokens.sql</files>
  <action>
    Create migration with:

    **Table: fcm_tokens**
    - id: uuid PK (gen_random_uuid)
    - user_id: uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE
    - device_token: text NOT NULL
    - platform: text NOT NULL DEFAULT 'android'
    - created_at: timestamptz DEFAULT now()
    - updated_at: timestamptz DEFAULT now()
    - UNIQUE(user_id, device_token)

    **RLS:**
    - Enable RLS on fcm_tokens
    - Policy: "Users manage own tokens" FOR ALL USING (auth.uid() = user_id)

    **RPCs (all SECURITY DEFINER):**

    1. `upsert_fcm_token(p_device_token text, p_platform text DEFAULT 'android')` → void
       - INSERT INTO fcm_tokens (user_id, device_token, platform) VALUES (auth.uid(), p_device_token, p_platform)
       - ON CONFLICT (user_id, device_token) DO UPDATE SET updated_at = now()
       - REVOKE FROM PUBLIC, GRANT TO authenticated

    2. `delete_fcm_token(p_device_token text)` → void
       - DELETE FROM fcm_tokens WHERE user_id = auth.uid() AND device_token = p_device_token
       - REVOKE FROM PUBLIC, GRANT TO authenticated

    3. `get_fcm_tokens_for_business(p_business_id uuid, p_roles text[] DEFAULT ARRAY['owner','admin'], p_exclude_user uuid DEFAULT NULL)` → TABLE(user_id uuid, device_token text)
       - Join fcm_tokens with auth.users on id = user_id
       - Filter: raw_user_meta_data->>'business_id' = p_business_id::text
       - Filter: raw_user_meta_data->>'role' = ANY(p_roles)
       - Filter: (raw_user_meta_data->>'active')::boolean IS TRUE
       - Filter: ft.user_id != COALESCE(p_exclude_user, '00000000-0000-0000-0000-000000000000'::uuid) <!-- audit-added: exclude caller from targets -->
       - REVOKE FROM PUBLIC (no GRANT — service_role only, used by Edge Function)

    **Follow existing REVOKE FROM PUBLIC pattern from migrations 015-021.**
  </action>
  <verify>SQL syntax valid, table structure consistent with existing schema patterns</verify>
  <done>AC-1, AC-2 data layer ready — tokens can be stored, queried, and deleted</done>
</task>

<task type="auto">
  <name>Task 2: Supabase Edge Function — send-notification</name>
  <files>supabase/functions/send-notification/index.ts</files>
  <action>
    <!-- audit-added: moved before checkpoint so code exists when user deploys -->
    Create Supabase Edge Function at `supabase/functions/send-notification/index.ts`:

    **Input:** POST body `{ event_type: string, data: object }`
    - event_type: 'new_order' | 'payment_collected' | 'discount_pending' | 'low_stock' | 'shift_opened' | 'shift_closed'
    - data: event-specific payload (driver name, store name, amount, product name, quantity)

    **Logic:**
    1. Verify Authorization header present (JWT from authenticated caller)
    2. Create Supabase client with SUPABASE_SERVICE_ROLE_KEY (env var auto-available in Edge Functions)
    3. Decode caller's JWT → get user info → extract business_id from user_metadata
    4. <!-- audit-added --> **Validate business_id is non-null.** If null or empty, return `{ error: "missing business_id in user metadata" }` with status 400. Do NOT proceed to query tokens with a null business_id.
    5. <!-- audit-added --> **Validate event_type against allowed values.** Check event_type is one of: 'new_order', 'payment_collected', 'discount_pending', 'low_stock', 'shift_opened', 'shift_closed'. If unknown, return `{ error: "unknown event_type" }` with status 400. Do NOT attempt to build notification content for unknown types.
    6. Determine target roles: most events → ['owner', 'admin'], discount_pending → ['owner']
    7. Call `get_fcm_tokens_for_business(p_business_id, p_roles, p_exclude_user)` RPC → get device tokens. **Pass caller's user_id as p_exclude_user** to prevent self-notifications. <!-- audit-added -->
    8. If no tokens → return `{ sent: 0 }`
    9. Generate FCM v1 OAuth2 access token:
       - Read FCM_SERVICE_ACCOUNT secret (Deno.env.get)
       - <!-- audit-added --> **If secret is missing or unparseable, return `{ error: "FCM not configured" }` with status 503.** Do not crash.
       - Build JWT: iss = client_email, sub = client_email, aud = "https://oauth2.googleapis.com/token", scope = "https://www.googleapis.com/auth/firebase.messaging"
       - Sign with RS256 using private_key from service account
       - POST to `https://oauth2.googleapis.com/token` with grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
       - Extract access_token from response
       - <!-- audit-added --> **Cache access token in module-level variable with expiry timestamp.** On each invocation, check if cached token exists and has >5 minutes remaining before expiry. If valid, reuse. If expired or absent, generate new. This works because Deno Deploy (Edge Functions runtime) preserves module state across warm invocations. Cold starts will regenerate.
    10. Build notification title + body (Arabic) based on event_type:

       | event_type | title | body pattern |
       |------------|-------|-------------|
       | new_order | طلب جديد | طلب جديد من {driver} في {store} |
       | payment_collected | تحصيل دفعة | {driver} حصّل {amount} د.ج من {store} |
       | discount_pending | طلب خصم | {driver} يطلب خصم {amount} د.ج في {store} |
       | low_stock | مخزون منخفض | {product}: بقي {quantity} فقط |
       | shift_opened | بداية وردية | {driver} بدأ وردية جديدة |
       | shift_closed | نهاية وردية | {driver} أنهى ورديته |

    11. Send to each token via FCM v1:
       - POST `https://fcm.googleapis.com/v1/projects/{project_id}/messages:send`
       - Headers: Authorization Bearer {access_token}, Content-Type application/json
       - Body: `{ message: { token, notification: { title, body }, data: { event_type, ...stringified data }, android: { priority: "high" } } }`
    12. Handle stale tokens: if FCM returns UNREGISTERED or NOT_FOUND, delete token from fcm_tokens
    13. Return `{ sent: N, failed: M }`

    **CORS:** Add standard Supabase CORS headers for browser/client invocation.

    **Key constraints:**
    - Use Deno crypto APIs for JWT RS256 signing (importPKCS8 + sign)
    - Service account key ONLY in Supabase secrets, never committed to repo
    - Handle all FCM API errors gracefully (log but don't crash)
  </action>
  <verify>

    - All 6 event_type values produce correct Arabic title/body
    - Unknown event_type returns 400
    - Null business_id returns 400
  </verify>
